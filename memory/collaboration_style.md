# 协作偏好与反馈

更新时间：2026-06-17

本文件记录与用户合作时观察到的、对未来对话有指导意义的偏好和反馈。

---

## 项目结构与代码风格

**偏好：模块化、合理分层，不接受单文件大杂烩。**

用户原话："不要把所有东西都挤在单文件里，要建好合理的项目结构，方便后期扩展"

**How to apply：** 做新功能时默认拆模块、定抽象接口（Protocol/ABC）、留扩展点（目录占位）。即使是早期阶段，也要把 device / reasoning 这种"未来会换实现"的层抽象出来。

---

**偏好：不接受硬编码全量映射表，但接受小而精的静态表 + 兜底策略。**

用户在 OPEN 启动 app 这件事上明确反对维护全量"中文名 → package"表，但接受"高频静态表 + 多级 fallback"的混合方案。

**How to apply：** 遇到"要不要维护表"的问题，默认走"小静态表 + 自动兜底"路线，不要一次性塞 100 条数据进表，也不要完全靠运行时探测。

---

## 执行授权与风险

**偏好：愿意把执行权交给 AI，但敏感动作期望先解释再做。**

- 用户多次说"你帮我跑"、"你来做"，对自动化操作有信任
- 但读 `api-key.txt` 时主动停下询问，用户明确表态同意 —— 这是对的做法
- 用户中断了"打开抖音"死循环跑飞的任务，然后说"你这玩意儿似乎彻底跑偏了方向"

**How to apply：**
- 普通命令（测试、安装、运行已写好的脚本）直接跑
- 触碰密钥 / 支付 / 远程仓库 / 批量删除前一律先问
- 跑出来发现走偏不要硬撑，主动停下做根因分析

---

**偏好：能自己跑通的事不要让用户做，但需要物理交互的（手机端弹窗确认）只能用户来。**

用户原话："看着比较复杂，你替我搞吧"

**How to apply：** 能 ssh / adb / CLI 解决的，默认我跑；只剩物理屏幕点击或浏览器登录这种，清晰告诉用户"这步只能你做"。

---

**重要：不要用工具删除文件（2026-06-08）**

即使是自己刚建的临时文件，也不要用工具删，告诉用户要删什么、由用户自己删。比"批量删除前先问"更强——是连单个删除动作都交给用户。

**Why：** 避免误删、数据丢失等不可逆操作，即使是临时文件也可能有价值。

**How to apply：** 创建临时探针、测试文件后，告诉用户位置和用途，由用户决定何时删除。

---

## 调试与问题分析

**偏好：出 bug 时希望做根因分析，不是反复重试。**

死循环 bug 出来后，用户没要求"再试一次"，而是希望说清楚"为什么"。列出 4 步根因链（模型给海外包名 → fuzzy 误匹配 → cache 固化 → 无死循环检测），用户接受这个分析方式。

**How to apply：** bug 报告里先讲机制（why），再讲修复（what）。不要"我改了 X，你再试"。

---

**偏好：细节决策（参数阈值、字符串长度门槛）可以直接定，但要解释取舍。**

fuzzy_match 阈值从 0.6 改 0.75、字符长度从 3 改 5 再回 4，用户全程没反对，只看结果。

**How to apply：** 这类微调直接做，但 commit message 或对话里说清楚"为什么是这个数"。

---

## 工具协作

**偏好：用 plan mode 做非平凡设计，但不要为小改动 plan。**

设计 phone_agent 项目结构时进 plan mode，用户配合得很好，中间用 AskUserQuestion 拿到关键决策。后续小改动（thinking_mode、修 bug）直接做，不再 plan。

**How to apply：** 涉及多文件、有架构决策、有多种合理方案的工作走 plan mode；单文件 / 单逻辑修改直接动手。

---

**偏好：Commit 消息中文，简洁但要点齐全，不要太啰嗦。**

看仓库历史风格相对随意，但用户认可结构化的 commit message（分要点 + Co-Authored-By）。

**How to apply：** 正式提交用结构化中文 message，WIP 存档可以简短。

---

## 不要做

- **不要静默装重型依赖到全局 Python**，装前提一下
- **不要假定模型输出格式稳定**，海外 / 国内、新版 / 旧版、JSON / markdown 都要兼容
- **不要自己删文件**：即使是自己刚建的临时文件，也告诉用户要删什么、由用户自己删
- **不要在对话中说"MVP"**：产品已经是正式产品，不是最小可行产品阶段
